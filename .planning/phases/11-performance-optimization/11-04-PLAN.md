# Phase 11 Plan 04 — DB Pool + Performance Indexes

## Goal

`core/database_pool.py` already implements a `DatabaseConnectionPool` singleton with `QueuePool(pool_size=5, max_overflow=10)`. The research identified "5 call sites with create_engine" in worker files, but audit reveals workers (`chat_tasks.py`, `gardener_tasks.py`, `helpers.py`) already use `db_pool`. The remaining gap is that `media_tasks.py` and `data_tasks.py` update document status via HTTP callbacks to the FastAPI API (not direct DB) — this is acceptable but should be verified. The larger win is adding missing DB indexes: no index on `documents(user_id)` or `chat_sessions(user_id, updated_at)` means every session list query does a full table scan.

## Tasks

### Task 1: Audit and verify all worker DB call sites use `db_pool`
**File:** `chatSNP170226/backend/src/worker/` (all worker files)
**What:** Confirm there are no remaining `create_engine()` calls in worker code that bypass the pool. Document findings. Fix any stragglers.
**How:**
1. Search all worker Python files for `create_engine`, `sessionmaker`, `Session(bind=`, raw `psycopg2.connect`:
   ```bash
   grep -rn "create_engine\|sessionmaker\|psycopg2.connect" backend/src/worker/
   ```
2. If any found: replace with `from src.core.database_pool import db_pool` and `db_pool.execute_query(...)`.
3. Verify `media_tasks.py` uses `_update_document_status` from `helpers.py` which uses `db_pool` (line 289 of helpers.py confirmed).
4. Verify `data_tasks.py` (`run_sql_query`) only reads DB via Vanna's internal connection (separate concern — Vanna manages its own connection to PostgreSQL for schema introspection). Document that Vanna DB connection is out of scope for this pool.
5. Update pool `max_overflow` from 10 → 2 in `database_pool.py` (currently 10 is too generous for a 2-worker setup):
   ```python
   max_overflow=2,   # was 10; 2 workers × (5+2) = 14 max connections total
   ```

### Task 2: Create `get_worker_engine()` convenience function
**File:** `chatSNP170226/backend/src/core/database.py` (FastAPI async DB file) + `chatSNP170226/backend/src/core/database_pool.py`
**What:** Add a clearly-named top-level function `get_worker_engine()` so worker code has a single canonical import, avoiding direct instantiation of `DatabaseConnectionPool`.
**How:**
1. In `database_pool.py`, add at the bottom:
   ```python
   def get_worker_engine():
       """
       Return the shared SQLAlchemy Engine for sync Celery worker use.
       Use this instead of create_engine() in worker files.
       Pool config: QueuePool(pool_size=5, max_overflow=2, pool_recycle=3600).
       """
       return get_db_pool().get_engine()
   ```
2. Update any `db_pool.get_engine()` call sites to use `get_worker_engine()` for consistency.
3. Add module docstring note: "Do NOT import this in FastAPI async routes — use `core/db.py` async session factory instead."

### Task 3: Alembic migration — performance indexes
**File:** `chatSNP170226/backend/alembic/versions/` (new migration file)
**What:** Add indexes that speed up the most common list queries:
- `GET /sessions` → `chat_sessions WHERE user_id = ? ORDER BY updated_at DESC`
- `GET /documents` → `documents WHERE user_id = ? ORDER BY created_at DESC`
- Message lookup → `messages WHERE session_id = ? ORDER BY created_at`
**How:**
1. Generate migration:
   ```bash
   cd chatSNP170226/backend && alembic revision --autogenerate -m "perf_indexes_phase11"
   ```
2. Edit the generated file to use `CONCURRENTLY` to avoid locking the table during creation:
   ```python
   from alembic import op

   def upgrade():
       op.execute(
           "CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_documents_user_id "
           "ON documents (user_id)"
       )
       op.execute(
           "CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_documents_user_created "
           "ON documents (user_id, created_at DESC)"
       )
       op.execute(
           "CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_chat_sessions_user_updated "
           "ON chat_sessions (user_id, updated_at DESC)"
       )
       op.execute(
           "CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_messages_session_created "
           "ON messages (session_id, created_at)"
       )

   def downgrade():
       op.execute("DROP INDEX CONCURRENTLY IF EXISTS idx_documents_user_id")
       op.execute("DROP INDEX CONCURRENTLY IF EXISTS idx_documents_user_created")
       op.execute("DROP INDEX CONCURRENTLY IF EXISTS idx_chat_sessions_user_updated")
       op.execute("DROP INDEX CONCURRENTLY IF EXISTS idx_messages_session_created")
   ```
3. Note: `CREATE INDEX CONCURRENTLY` cannot run inside a transaction. Add `connection.execute(text("COMMIT"))` before, or configure the migration to run outside a transaction block:
   ```python
   # At top of migration file:
   # revision = '...'
   # down_revision = '...'
   # transactional_ddl = False  # Alembic 1.x: use connection_kwarg instead
   ```
   Use `op.get_context().autocommit_block()` or set `transaction_per_migration = false` in `env.py`.

### Task 4: Tests — pool reuse, index existence, query plan
**File:** `chatSNP170226/backend/tests/test_db_pool_indexes.py` (new)
**What:** Verify pool is reused across calls and indexes exist in the DB.
**How:**
1. `test_pool_singleton_same_engine`: call `get_worker_engine()` twice; assert `id(engine1) == id(engine2)`.
2. `test_pool_max_overflow_set`: inspect `engine.pool.size()` and `engine.pool._max_overflow`; assert values match config.
3. `test_indexes_exist` (integration, requires running DB):
   ```python
   def test_indexes_exist(db_engine):
       with db_engine.connect() as conn:
           result = conn.execute(text(
               "SELECT indexname FROM pg_indexes WHERE tablename IN "
               "('documents','chat_sessions','messages') AND indexname LIKE 'idx_%'"
           )).fetchall()
           index_names = {r[0] for r in result}
           assert "idx_documents_user_created" in index_names
           assert "idx_chat_sessions_user_updated" in index_names
   ```
4. `test_query_plan_uses_index` (integration):
   ```python
   def test_query_plan_uses_index(db_engine):
       with db_engine.connect() as conn:
           plan = conn.execute(text(
               "EXPLAIN SELECT * FROM documents WHERE user_id='test' ORDER BY created_at DESC LIMIT 20"
           )).fetchall()
           plan_text = " ".join(str(r) for r in plan)
           assert "Index Scan" in plan_text or "Bitmap Index Scan" in plan_text
   ```

## Success Criteria
- [ ] `grep -rn "create_engine" backend/src/worker/` returns zero results (excluding database_pool.py itself)
- [ ] `get_worker_engine()` returns the same object on repeated calls (singleton verified)
- [ ] All 4 indexes exist in production DB after running `alembic upgrade head`

- [ ] DB connection count under load stays ≤ 14 (2 workers × 7 max connections)

## Threat Model / Risks
- **`CREATE INDEX CONCURRENTLY` outside transaction:** Must configure Alembic migration to not wrap in a transaction. If this fails, fall back to non-concurrent CREATE (with a maintenance window notice).
- **Autogenerate detects spurious changes:** `alembic revision --autogenerate` may pick up unrelated model drift. Review generated migration carefully before committing.
- **Vanna's internal DB connection bypasses pool:** Vanna uses its own SQLAlchemy engine for schema introspection (read-only). This is acceptable — Vanna connections are infrequent and read-only. Document as known exception.
- **pool_size=5 per worker × 2 workers = 10 connections minimum:** PostgreSQL `max_connections` default is 100; this is well within limits. `max_overflow=2` caps burst at 14 total.
