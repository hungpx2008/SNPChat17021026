# Phase 11 Plan 02 — Celery Worker Hardening

## Goal

The `worker_media` service runs with `-c 1` and no per-task time limits, meaning a single stuck upload blocks all subsequent uploads indefinitely. This plan bumps concurrency to 2, adds `--soft-time-limit` / `--time-limit` flags to the worker command, declares `soft_time_limit` / `time_limit` on each task decorator (belt-and-suspenders), and adds a `/health` endpoint to the mem0-service (which is now a local lib) plus a pre-flight ping inside `process_document` that fails fast if mem0 is unreachable.

### Task 2: Add `soft_time_limit` / `time_limit` to `process_document` task decorator
**File:** `chatSNP170226/backend/src/worker/media_tasks.py`
**What:** Task decorator at line 62 currently has `bind=True, max_retries=2` but no time limits. Worker-level limits kill the process, but per-task limits raise `SoftTimeLimitExceeded` which allows graceful cleanup.
**How:**
1. Update decorator:
   ```python
   @celery_app.task(
       name="src.worker.tasks.process_document",
       bind=True,
       max_retries=2,
       soft_time_limit=300,
       time_limit=360,
   )
   ```
2. Add `except SoftTimeLimitExceeded` handler in the task body:
   ```python
   from celery.exceptions import SoftTimeLimitExceeded
   ...
   except SoftTimeLimitExceeded:
       logger.error(f"[process] Tác vụ vượt quá 300s — hủy xử lý {filename}")
       _update_document_status(document_id, "error", user_id,
           error_msg="Xử lý tài liệu quá thời gian cho phép (>5 phút). Vui lòng thử lại với file nhỏ hơn.")
       raise
   ```
3. Ensure `_update_document_status` is called before re-raising so the DB row reflects `error` status.

### Task 5: Mem0 health pre-flight check + circuit breaker in `process_document`
**File:** `chatSNP170226/backend/src/worker/media_tasks.py` + `chatSNP170226/backend/src/core/mem0_local.py`
**What:** Since mem0 is now a local library (not a separate HTTP service per `mem0_local.py`), the health check becomes verifying the singleton is initialized and Qdrant is reachable, rather than pinging an HTTP endpoint.
**How:**
1. Add `health_check()` function to `mem0_local.py`:
   ```python
   def mem0_health_check(timeout_s: float = 5.0) -> bool:
       """Return True if mem0 singleton and Qdrant are reachable."""
       import concurrent.futures
       def _check():
           m = get_mem0()  # triggers lazy init
           # Attempt a lightweight Qdrant ping via mem0's vector store
           m.vector_store.client.get_collections()
           return True
       try:
           with concurrent.futures.ThreadPoolExecutor(max_workers=1) as ex:
               future = ex.submit(_check)
               return future.result(timeout=timeout_s)
       except Exception as exc:
           logger.warning(f"[mem0-health] Check failed: {exc}")
           return False
   ```
2. In `process_document`, before entering the processing pipeline, add a circuit breaker with 3 attempts:
   ```python
   from src.core.mem0_local import mem0_health_check
   import time as _time

   for attempt in range(1, 4):
       if mem0_health_check(timeout_s=5.0):
           break
       logger.warning(f"[process] Mem0 health check failed (attempt {attempt}/3)")
       if attempt == 3:
           _update_document_status(document_id, "error", user_id,
               error_msg="Dịch vụ nhúng văn bản không khả dụng. Vui lòng thử lại sau.")
           return {"status": "error", "reason": "mem0_unhealthy"}
       _time.sleep(2 ** attempt)  # 2s, 4s
   ```
3. Log total pre-flight time via `log_stage(document_id, "preflight", elapsed_ms)` (uses helper from 11-01).

## Threat Model / Risks
- **`-c 2` doubles RAM for Docling:** DocumentConverter weights are per-process. Two workers = 2× model RAM. Mitigated by `mem_limit: 2g` docker constraint and `--max-tasks-per-child=50`.
- **`SoftTimeLimitExceeded` not caught in nested libs:** Docling or Whisper may catch `BaseException` internally. The worker-level `--time-limit` (SIGKILL) is the ultimate backstop.
- **mem0 local lib Qdrant ping adds latency:** Only 3–5s on failure path; on success path, singleton is already initialized so `get_mem0()` returns immediately.
- **`run_sql_query` 60s limit too tight for complex queries:** Adjustable via env var `SQL_TASK_SOFT_LIMIT` if needed. Start at 60s and tune from production metrics.
