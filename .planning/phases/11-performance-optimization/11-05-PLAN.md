# Phase 11 Plan 05 — Chat History Pagination + HTTP Pool Verification + Regression Tests

## Goal

Chat sessions accumulate unbounded message history. The existing `GET /sessions/{id}/messages?limit=N` endpoint supports a `limit` parameter (line 74 of `chat.py`) but the frontend loads all messages at once. This plan adds cursor-based pagination (`before_id` parameter), updates the frontend to load last 50 on open with scroll-to-top lazy loading, verifies the Phase 10 `http_client` singleton is consistently used (no stray `httpx.Client()` instantiations), and creates a performance regression test suite to gate future releases.

## Tasks

### Task 1: Backend — paginate `GET /sessions/{id}/messages` with `before_id` cursor
**File:** `chatSNP170226/backend/src/api/chat.py`
**What:** The messages endpoint at line 74 accepts `limit` but no cursor. Add `before_id: str | None` query param for cursor-based pagination so the frontend can lazily load older messages.
**How:**
1. Update the route signature:
   ```python
   @router.get("/{session_id}/messages")
   async def get_messages(
       session_id: str,
       limit: int = Query(default=50, ge=1, le=200),
       before_id: str | None = Query(default=None),
       db: AsyncSession = Depends(get_db),
       ...
   ):
   ```
2. In `MessageRepository` (or inline), add `get_messages_paginated(session_id, limit, before_id)`:
   ```python
   async def get_messages_paginated(session_id, limit=50, before_id=None):
       query = select(Message).where(Message.session_id == session_id)
       if before_id:
           # Get created_at of the before_id message, then fetch messages older than it
           ref = await session.get(Message, before_id)
           if ref:
               query = query.where(Message.created_at < ref.created_at)
       query = query.order_by(Message.created_at.desc()).limit(limit)
       result = await session.execute(query)
       messages = result.scalars().all()
       return list(reversed(messages))  # Return in chronological order
   ```
3. Response envelope:
   ```json
   {
     "messages": [...],
     "has_more": true,
     "oldest_id": "uuid-of-first-message-in-this-page"
   }
   ```
   `has_more = len(messages) == limit` (if we got a full page, there may be more).
4. Keep backward compat: when `before_id` is None and `limit` is None, return last 50 messages (existing callers unchanged).

### Task 2: Frontend — load last 50 messages, lazy-load older via scroll-to-top
**File:** `chatSNP170226/frontend/src/components/` (chat message list component)
**What:** Currently all messages are loaded at session open. Update to load only last 50, with a "Load earlier messages" trigger when user scrolls to top.
**How:**
1. Create/update `useMessages(sessionId)` hook:
   ```typescript
   // hooks/useMessages.ts
   export function useMessages(sessionId: string) {
     const [messages, setMessages] = useState<Message[]>([]);
     const [hasMore, setHasMore] = useState(false);
     const [oldestId, setOldestId] = useState<string | null>(null);
     const [loadingMore, setLoadingMore] = useState(false);

     const loadInitial = async () => {
       const res = await fetch(`/api/sessions/${sessionId}/messages?limit=50`);
       const data = await res.json();
       setMessages(data.messages);
       setHasMore(data.has_more);
       setOldestId(data.oldest_id);
     };

     const loadMore = async () => {
       if (!hasMore || loadingMore || !oldestId) return;
       setLoadingMore(true);
       const res = await fetch(`/api/sessions/${sessionId}/messages?limit=50&before_id=${oldestId}`);
       const data = await res.json();
       setMessages(prev => [...data.messages, ...prev]);  // prepend older msgs
       setHasMore(data.has_more);
       setOldestId(data.oldest_id ?? null);
       setLoadingMore(false);
     };

     useEffect(() => { loadInitial(); }, [sessionId]);
     return { messages, hasMore, loadMore, loadingMore };
   }
   ```
2. In the message list component, add scroll handler:
   ```typescript
   const handleScroll = (e: React.UIEvent<HTMLDivElement>) => {
     if (e.currentTarget.scrollTop === 0 && hasMore) {
       loadMore();
     }
   };
   ```
3. When `loadMore` prepends messages, preserve scroll position (save `scrollHeight` before update, restore `scrollHeight - prevScrollHeight` after).
4. Show `loadingMore` spinner at top of list; when `!hasMore`, show "Đây là tin nhắn đầu tiên trong cuộc trò chuyện." label.
5. Remove any existing `?limit=1000` or unlimited fetches in session-open code paths.

**Part B — Regression test suite:**
1. `test_upload_small_pdf_under_90s`:
   ```python
   import time, httpx, pytest

   @pytest.mark.integration
   @pytest.mark.slow
   def test_upload_small_pdf_under_90s(api_base_url, test_user_token, sample_pdf_path):
       """Upload a ≤2MB PDF and assert processing completes within 90 seconds."""
       client = httpx.Client(base_url=api_base_url, timeout=120.0)
       # Step 1: Upload file
       t0 = time.monotonic()
       with open(sample_pdf_path, "rb") as f:
           resp = client.post("/upload", files={"file": f},
               headers={"Authorization": f"Bearer {test_user_token}"})
       assert resp.status_code == 200
       document_id = resp.json()["document_id"]

       # Step 2: Poll until ready or timeout
       deadline = t0 + 90
       while time.monotonic() < deadline:
           status_resp = client.get(f"/documents/{document_id}")
           if status_resp.json().get("status") == "ready":
               break
           time.sleep(2)
       else:
           pytest.fail(f"Upload did not complete within 90s (document_id={document_id})")

       elapsed = time.monotonic() - t0
       assert elapsed <= 90, f"Upload took {elapsed:.1f}s, expected ≤90s"
   ```
2. `test_chat_response_under_5s`:

3. `test_two_concurrent_uploads_dont_block`:

   ```python
   @pytest.mark.integration
   @pytest.mark.slow
   def test_two_concurrent_uploads_dont_block(api_base_url, test_user_token, sample_pdf_path):
       """Two simultaneous uploads should both complete within 120s."""
       import concurrent.futures
       def upload():
           with open(sample_pdf_path, "rb") as f:
               return httpx.post(f"{api_base_url}/upload",
                   files={"file": (f.name, f)},
                   headers={"Authorization": f"Bearer {test_user_token}"},
                   timeout=30.0).json()["document_id"]

       t0 = time.monotonic()
       with concurrent.futures.ThreadPoolExecutor(max_workers=2) as ex:
           doc_ids = list(ex.map(lambda _: upload(), range(2)))

       # Poll both until ready
       deadline = t0 + 120
       client = httpx.Client(base_url=api_base_url, timeout=30.0)
       while time.monotonic() < deadline:
           statuses = [client.get(f"/documents/{did}").json().get("status") for did in doc_ids]
           if all(s == "ready" for s in statuses):
               break
           time.sleep(3)
       else:
           pytest.fail("One or both concurrent uploads did not complete within 120s")
   ```

## Threat Model / Risks
- **Cursor pagination race condition:** If new messages arrive between pages, `before_id` cursor based on `created_at` may cause duplicates if two messages share the same timestamp. Use message UUID as cursor (already implemented via `before_id` pointing to message ID, not timestamp).
- **Scroll position restoration complexity:** `scrollHeight` delta technique works in most browsers but may jitter on Safari. Use `scrollTop` save/restore wrapped in `requestAnimationFrame`.
- **Regression tests require running stack:** Mark as `@pytest.mark.integration` and exclude from unit test CI run. Run separately in a dedicated integration test stage.
- **`httpx.Client` singleton across Celery workers:** The singleton is process-local. Each worker process has its own `_client` instance (expected behavior). No sharing across processes.
- **Test PDF fixture size:** Bundled test PDF must be ≤ 2MB and have at least 5 pages to exercise chunking. Use a sanitized version of `Biểu giá dịch vụ.pdf` or generate a synthetic PDF in the fixture.
