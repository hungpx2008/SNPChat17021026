# Plan 04: Polish — Error Handling, Docstrings, Type Safety, Hook APIs, i18n

## Goal
Replace generic `except Exception` with specific exceptions in high-value worker files (34 instances), add docstrings to all newly created and modified modules, remove all `any` type annotations from frontend code, and encapsulate hook APIs to prevent raw state leaking.

## must_haves
- `except Exception` count in `chat_tasks.py` + `media_tasks.py` + `data_tasks.py` reduced by at least 60% (from 34 to < 14)
- All functions in `rag/` subpackage and `docling/` subpackage have docstrings
- `: any` count across frontend is 0 (was 16)
- `use-chat-messages.ts` does NOT export `setMessages` — only semantic actions like `addMessage`, `updateMessage`, `clearMessages`
- `use-chat-sessions.ts` does NOT export `setChatHistory` or `setActiveChatId` — only semantic actions

## threat_model
- **Exception narrowing risk**: Replacing `except Exception` with specific types could miss edge cases from unstable third-party libs (Docling, Mem0). Mitigation: for third-party library calls, use a typed catch + a final `except Exception as e: logger.warning("Unexpected %s: %s", type(e).__name__, e)` as safety net — but log at WARNING, not silently swallow.
- **Hook API change**: Changing return types of hooks breaks all consumers. All components that use `setMessages` directly must be updated in the same commit. Verify all consumers by grepping before changing.
- **Type narrowing**: Replacing `any` with `unknown` or specific types may cause TypeScript compile errors. Run `npx tsc --noEmit` after changes.
